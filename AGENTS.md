# AGENTS.md

本文件用于规范 `batt_soh` 仓库中的人机协作方式，面向开发者与 AI 代理共同使用。

## 1. 适用对象与目标

- 适用对象：人类开发者 + AI 代理。
- 协作目标：在保证可追溯、安全、可维护的前提下，完成电池 SOH 相关的数据处理、训练与评估工作。

## 2. 默认工作方式

- 默认采用“先计划，后执行”模式。
- AI 在改动代码/文件前，应先给出简要计划（做什么、影响哪些文件、预期结果）。
- 计划经用户确认后再执行改动；若用户明确要求“直接做”，可直接执行。

## 3. 命令与操作权限

- 读取类操作可直接执行（如查看目录、读取文件、检索文本、查看日志）。
- 除读取外的操作一律先确认后执行，包括但不限于：
- 新建/修改/删除文件
- 运行训练、测试、批处理脚本
- 安装依赖、下载数据、执行可能耗时命令
- 任何潜在破坏性操作（覆盖、清理、重置等）必须单独明确确认。

### 3.1 Python 解释器约定

- 本仓库统一使用 `C:\Users\pal\pyenv\colab` 作为默认 Python 运行环境。
- 推荐调用方式：在 `C:\Users\pal\pyenv\colab` 目录执行 `pipenv run python <script>`。
- 若需直接指定解释器路径，以该环境实际解析出的解释器为准（例如 `...\Scripts\python.exe`）。
- 未经用户明确确认，不切换到其他解释器环境（如 `ds_env`）。

## 4. 输出与沟通风格

- 输出应“结果 + 解释”并重，便于团队理解与复现。
- 对每次关键改动说明：
- 改了什么
- 为什么这么改
- 影响范围
- 如何验证
- 对不确定项要显式标注假设，避免隐式推断。

## 5. 项目核心任务优先级

当前项目核心任务包含以下三类，默认都要覆盖：

- 数据清洗与特征工程
- SOH 建模训练
- 训练评估框架搭建

执行时遵循“先打通最小可用链路，再逐步增强”的策略。

## 6. 硬约束（必须遵守）

- 不新建外层工程目录，所有内容直接在仓库现有结构下组织。
- 数据相关说明文档优先放在 `data/raw` 邻近位置（如数据字典、字段说明）。
- 文件命名使用英文小写与蛇形风格（`snake_case`）为默认规范。
- 默认先交付最小可用版本（MVP），再进行迭代优化。

## 7. 数据约定（当前已知）

- 列结构当前一致，字段定义参考：
- `data/raw/DATA_DICTIONARY.md`

## 9. 常用任务模板

以下模板用于快速对齐“输入-执行-输出-验收”。

### 模板 A：新增数据集（raw 数据接入）

- 目标：将新原始数据接入仓库并完成最小可用的数据说明。
- 输入：
- 数据来源与格式（如 `cycles_*.csv`）
- 存放路径（默认 `data/raw/<dataset_name>`）
- 关键字段说明（若已知）
- 标准步骤：
- 读取并检查文件可访问性与命名规范
- 抽样检查列结构（避免直接全量重处理）
- 对比现有 schema（字段名、类型、单位、缺失情况）
- 更新或新增数据字典文档（优先放 `data/raw` 邻近）
- 输出：
- 新数据存放路径
- schema 检查结果（是否兼容现有流程）
- 文档变更清单（如 `DATA_DICTIONARY.md` 更新项）
- 验收标准：
- 数据可被脚本读取
- 字段定义清晰、单位明确
- 不破坏现有目录组织规则

### 模板 B：训练一次基线模型（Baseline）

- 目标：在当前数据上完成一次可复现的最小训练闭环。
- 输入：
- 数据范围（策略/电芯/循环范围）
- 基线模型与核心参数（学习率、batch size、epoch）
- 输出路径约定（`outputs/checkpoints`, `outputs/logs`）
- 标准步骤：
- 明确训练计划与影响文件，先确认再执行
- 准备最小训练配置（优先可跑通）
- 执行训练并保存日志与权重
- 记录关键指标与异常（如收敛失败、数值异常）
- 输出：
- 训练配置摘要
- 模型权重路径
- 关键训练指标（loss/误差等）
- 验收标准：
- 训练流程可复现
- 输出文件完整可追踪
- 指标可用于后续横向比较

### 模板 C：生成评估报告（对比实验）

- 目标：对一个或多个模型在同一评估集上形成可阅读结论。
- 输入：
- 待评估模型列表
- 固定评估数据范围与指标定义
- 报告输出位置与格式（Markdown 优先）
- 标准步骤：
- 固化评估口径（数据切分、指标、阈值）
- 执行评估并收集结果
- 对比各模型表现与失败样例
- 给出下一步优化建议（数据/特征/模型）
- 输出：
- 评估结果表（核心指标）
- 结论摘要（谁更优、在什么条件下更优）
- 后续行动建议（最多 3 条）
- 验收标准：
- 评估口径一致且可复现
- 结论有数据支撑
- 报告可直接用于讨论和决策

## 10. 文档目录结构及用途

文档相关内容按“就近、清晰、可检索”原则组织，建议如下：

- `data/raw/`：原始数据存放区，只做只读留存与必要抽样检查，不直接覆盖源数据。
- `data/raw/DATA_DICTIONARY.md`：原始数据字段字典（中英文名、单位、字段说明、注意事项）。
- `data/processed/`：清洗、对齐、特征化后的可训练数据（由流程生成，可再生）。
- `data/sample/`：小样本数据集，用于调试与快速验证（优先用于冒烟测试）。
- `configs/`：训练、评估、数据处理配置文件（参数来源应集中管理在此目录）。
- `notebooks/`：探索分析与实验记录（EDA、可视化、临时验证），结论应回写到正式文档或代码。
- `src/`：核心源码目录（模型、数据处理、工具函数等可复用逻辑）。
- `train/`：训练入口脚本与训练流程编排代码。
- `test/`：测试/评估入口脚本与验证逻辑。
- `scripts/`：命令行脚本（如一键训练、评估、数据处理辅助脚本）。
- `outputs/checkpoints/`：训练得到的模型权重与检查点文件。
- `outputs/logs/`：训练与评估过程日志（运行证据，不作为规范文档）。
- `outputs/predictions/`：模型输出结果与推理产物（用于分析与复核）。
- `README_AI_STRUCTURE.md`：项目 AI 训练/测试目录结构说明（面向快速上手）。
- `AGENTS.md`：人机协作规范、执行边界、任务模板与约定。

文档维护规则：

- 新增字段或单位变化时，优先更新 `data/raw/DATA_DICTIONARY.md`。
- 数据处理流程或产物口径变化时，更新 `data/processed/` 对应说明（如新增 README 或任务记录）。
- 调试样本口径变化时，更新 `data/sample/` 说明，确保样本可复现。
- 配置项或默认参数变化时，更新 `configs/` 中对应配置并在变更说明中标注影响范围。
- notebook 产出若形成稳定结论，需沉淀到 `src/`、`configs/` 或正式文档，避免仅保留在 notebook。
- 流程、协作规则变化时，更新 `AGENTS.md`。
- 目录结构变化时，更新 `README_AI_STRUCTURE.md`。
- 每次重要文档更新，建议在任务反馈中附“变更摘要 + 生效范围”。

## 13. 补充规则

### 13.1 技术栈约定

- 深度学习建模任务必须使用 `PyTorch`。
- 传统机器学习及统计任务可以使用 `sklearn`等工具，但新导入包时必须先让用户确认。
- 数据处理使用 `pandas` / `numpy`。
- 可视化使用 `matplotlib` / `seaborn`。
- 所有预估耗时较长的任务必须增加epoch日志实时落盘和追踪。
- 所有机器学习或深度学习的任务均需要在训练架构时考虑到对colab环境的兼容，避免移植到云端训练时的工作量。

### 13.2 代码规范

- 所有函数必须包含类型标注（typing）。
- 所有函数必须包含 docstring。
- 禁止提交伪代码，交付代码必须可运行。

### 13.3 训练与评估最低要求

- 默认任务方向：电池数据特征工程及容量预测。
- 评估结果必须输出：`MSE`、`R2`。
- 评估过程必须包含图表：预测值 vs 真实值散点图、训练 loss 曲线。

### 13.4 禁止行为

- 不允许随意修改现有接口。
- 不允许跳过异常处理逻辑。

### 13.5 交付输出要求

- 必须给出完整代码。
- 必须给出调用示例。
- 必须可直接运行。

### 13.6 行为模式

- 优先修改现有代码，而不是重写。
- 优先以最小改动实现目标。

### 13.7 Colab 长任务验收门槛

凡是修改训练入口、Colab notebook、输出目录规则、checkpoint、resume 逻辑、target pack、模型输入/输出维度或会影响云端训练成本的参数，必须先完成本地或小样本 `colab_contract_smoke` 验收，再建议用户在 Colab 执行正式长任务。

最低验收项：

- CLI 契约检查：`--help` 可运行；默认参数、显式参数优先级、关键枚举值（如 `target_pack`）符合预期。
- Notebook 命令一致性检查：notebook 生成的命令、脚本实际打印的输出目录、后续检查 cell 读取的目录必须一致。
- 路径规则检查：显式传入的 `--output-dir` / `--full-refresh-output-dir` 优先于脚本自动命名；禁止重复追加 `_smoke`、`_full` 等后缀。
- Target 维度检查：修改 dQdV/retention 目标后，必须扫描代码、notebook、Markdown、报告模板，确认旧目标名、旧维度描述、旧输出列检查没有残留。
- 最小链路检查：必须跑通 smoke tune、smoke full-refresh，并确认关键产物存在，包括 metrics、predictions、run_config、report、loss 曲线、checkpoint。
- 恢复训练检查：涉及 checkpoint 或 resume 的改动，必须用已有或构造的 `latest.pt` 验证 `--resume-interrupted` / `--full-refresh-resume-interrupted` 能恢复，且日志明确打印恢复 epoch。
- 版本兼容检查：checkpoint 读写必须考虑 Colab 当前 PyTorch 行为，尤其是 `torch.load(weights_only=...)` 兼容性；不得把恢复训练错误留给正式 Colab 任务暴露。
- 输出检查：smoke 产物中必须验证目标维度、关键模型名、关键指标列和预测列是否与配置一致。

默认执行顺序：

```text
接口契约稳定 -> 路径与恢复机制稳定 -> smoke 产物闭环 -> 正式训练
```

禁止以“先上云跑正式任务，再根据报错补修”的方式处理高成本训练链路。若因数据规模或环境限制无法完整执行上述验收，必须在反馈中明确说明未验证项、风险和建议的最小替代验证。
