# AGENTS.md

本文件用于规范 `batt_soh` 仓库中的人机协作方式，面向开发者与 AI 代理共同使用。

## 1. 适用对象与目标

- 适用对象：人类开发者 + AI 代理。
- 协作目标：在保证可追溯、安全、可维护的前提下，完成电池 SOH 相关的数据处理、训练与评估工作。

## 2. 默认工作方式

- 默认采用“先计划，后执行”模式。
- AI 在改动代码/文件前，应先给出简要计划（做什么、影响哪些文件、预期结果）。
- 计划经用户确认后再执行改动；若用户明确要求“直接做”，可直接执行。

## 3. 命令与操作权限

- 读取类操作可直接执行（如查看目录、读取文件、检索文本、查看日志）。
- 除读取外的操作一律先确认后执行，包括但不限于：
- 新建/修改/删除文件
- 运行训练、测试、批处理脚本
- 安装依赖、下载数据、执行可能耗时命令
- 任何潜在破坏性操作（覆盖、清理、重置等）必须单独明确确认。

### 3.1 Python 解释器约定

- 本仓库默认使用 `C:\Users\pal\pyenv\ds_env` 对应的 Python 运行环境。
- 推荐调用方式：在 `C:\Users\pal\pyenv\ds_env` 目录执行 `pipenv run python <script>`。
- 若需直接指定解释器路径，以该环境解析出的解释器为准（例如 `...\Scripts\python.exe`）。

## 4. 输出与沟通风格

- 输出应“结果 + 解释”并重，便于团队理解与复现。
- 对每次关键改动说明：
- 改了什么
- 为什么这么改
- 影响范围
- 如何验证
- 对不确定项要显式标注假设，避免隐式推断。

## 5. 项目核心任务优先级

当前项目核心任务包含以下三类，默认都要覆盖：

- 数据清洗与特征工程
- SOH 建模训练
- 训练评估框架搭建

执行时遵循“先打通最小可用链路，再逐步增强”的策略。

## 6. 硬约束（必须遵守）

- 不新建外层工程目录，所有内容直接在仓库现有结构下组织。
- 数据相关说明文档优先放在 `data/raw` 邻近位置（如数据字典、字段说明）。
- 文件命名使用英文小写与蛇形风格（`snake_case`）为默认规范。
- 默认先交付最小可用版本（MVP），再进行迭代优化。

## 7. 数据约定（当前已知）

- `data/raw` 下存在大量 `cycles_*.csv` 原始文件。
- 列结构当前一致，字段定义参考：
- `data/raw/DATA_DICTIONARY.md`
- 已确认：`ts` 单位为秒（s）。

## 8. 变更与记录建议

- 每次任务结束建议附上简短变更记录：
- 修改文件清单
- 是否运行验证及验证结果
- 待确认问题/后续建议

## 9. 常用任务模板

以下模板用于快速对齐“输入-执行-输出-验收”。

### 模板 A：新增数据集（raw 数据接入）

- 目标：将新原始数据接入仓库并完成最小可用的数据说明。
- 输入：
- 数据来源与格式（如 `cycles_*.csv`）
- 存放路径（默认 `data/raw/<dataset_name>`）
- 关键字段说明（若已知）
- 标准步骤：
- 读取并检查文件可访问性与命名规范
- 抽样检查列结构（避免直接全量重处理）
- 对比现有 schema（字段名、类型、单位、缺失情况）
- 更新或新增数据字典文档（优先放 `data/raw` 邻近）
- 输出：
- 新数据存放路径
- schema 检查结果（是否兼容现有流程）
- 文档变更清单（如 `DATA_DICTIONARY.md` 更新项）
- 验收标准：
- 数据可被脚本读取
- 字段定义清晰、单位明确
- 不破坏现有目录组织规则

### 模板 B：训练一次基线模型（Baseline）

- 目标：在当前数据上完成一次可复现的最小训练闭环。
- 输入：
- 数据范围（策略/电芯/循环范围）
- 基线模型与核心参数（学习率、batch size、epoch）
- 输出路径约定（`outputs/checkpoints`, `outputs/logs`）
- 标准步骤：
- 明确训练计划与影响文件，先确认再执行
- 准备最小训练配置（优先可跑通）
- 执行训练并保存日志与权重
- 记录关键指标与异常（如收敛失败、数值异常）
- 输出：
- 训练配置摘要
- 模型权重路径
- 关键训练指标（loss/误差等）
- 验收标准：
- 训练流程可复现
- 输出文件完整可追踪
- 指标可用于后续横向比较

### 模板 C：生成评估报告（对比实验）

- 目标：对一个或多个模型在同一评估集上形成可阅读结论。
- 输入：
- 待评估模型列表
- 固定评估数据范围与指标定义
- 报告输出位置与格式（Markdown 优先）
- 标准步骤：
- 固化评估口径（数据切分、指标、阈值）
- 执行评估并收集结果
- 对比各模型表现与失败样例
- 给出下一步优化建议（数据/特征/模型）
- 输出：
- 评估结果表（核心指标）
- 结论摘要（谁更优、在什么条件下更优）
- 后续行动建议（最多 3 条）
- 验收标准：
- 评估口径一致且可复现
- 结论有数据支撑
- 报告可直接用于讨论和决策

## 10. 文档目录结构及用途

文档相关内容按“就近、清晰、可检索”原则组织，建议如下：

- `data/raw/`：原始数据存放区，只做只读留存与必要抽样检查，不直接覆盖源数据。
- `data/raw/DATA_DICTIONARY.md`：原始数据字段字典（中英文名、单位、字段说明、注意事项）。
- `data/processed/`：清洗、对齐、特征化后的可训练数据（由流程生成，可再生）。
- `data/sample/`：小样本数据集，用于调试与快速验证（优先用于冒烟测试）。
- `configs/`：训练、评估、数据处理配置文件（参数来源应集中管理在此目录）。
- `notebooks/`：探索分析与实验记录（EDA、可视化、临时验证），结论应回写到正式文档或代码。
- `src/`：核心源码目录（模型、数据处理、工具函数等可复用逻辑）。
- `train/`：训练入口脚本与训练流程编排代码。
- `test/`：测试/评估入口脚本与验证逻辑。
- `scripts/`：命令行脚本（如一键训练、评估、数据处理辅助脚本）。
- `outputs/checkpoints/`：训练得到的模型权重与检查点文件。
- `outputs/logs/`：训练与评估过程日志（运行证据，不作为规范文档）。
- `outputs/predictions/`：模型输出结果与推理产物（用于分析与复核）。
- `README_AI_STRUCTURE.md`：项目 AI 训练/测试目录结构说明（面向快速上手）。
- `AGENTS.md`：人机协作规范、执行边界、任务模板与约定。

文档维护规则：

- 新增字段或单位变化时，优先更新 `data/raw/DATA_DICTIONARY.md`。
- 数据处理流程或产物口径变化时，更新 `data/processed/` 对应说明（如新增 README 或任务记录）。
- 调试样本口径变化时，更新 `data/sample/` 说明，确保样本可复现。
- 配置项或默认参数变化时，更新 `configs/` 中对应配置并在变更说明中标注影响范围。
- notebook 产出若形成稳定结论，需沉淀到 `src/`、`configs/` 或正式文档，避免仅保留在 notebook。
- 流程、协作规则变化时，更新 `AGENTS.md`。
- 目录结构变化时，更新 `README_AI_STRUCTURE.md`。
- 每次重要文档更新，建议在任务反馈中附“变更摘要 + 生效范围”。

## 11. 训练集与验证集划分结果

当前已完成基于 `policy + cell_code` 粒度的样本划分（每个组合视为一个样本），并结合以下信息进行分层与覆盖：

- policy 三元参数：初始倍率、转折 SOC、转折后倍率
- 电芯寿命指标：`max_cycles`（每个 `policy + cell_code` 的最大循环次数）

本次划分结果：

- 总样本数：`187`
- 训练集样本数：`135`
- 验证集样本数：`52`

对应文件：

- `data/processed/train_policy_cell_samples.csv`
- `data/processed/valid_policy_cell_samples.csv`

相关脚本：

- `scripts/split_train_valid_by_policy_cell.py`

划分说明（当前默认策略）：

- 优先从“同一 policy 下多 cell”的策略中抽取验证样本。
- 验证集覆盖变电流工况（`VARCHARGE_*`）。
- 验证集保留部分仅单 cell 的策略样本，避免样本类型偏置。
